# System Profile Creator

<GUARD>
**Skip this skill if ALL of these are true:**
- `/data/pi-agent/agents/skills/system-profile/SKILL.md` exists
- User did NOT explicitly ask to update/refresh/regenerate the profile

If system-profile exists and user didn't ask to update it, read that skill instead.
</GUARD>

## Purpose

This skill inspects the live Home Assistant instance and generates a `system-profile`
skill that documents this specific system's configuration structure. Every HA instance
is different — some use `!include` to split configs, others keep everything in
`configuration.yaml`. The generated profile tells future sessions exactly where things
are on THIS system.

## When to run

- **First session**: system-profile doesn't exist yet
- **Explicit request**: user says "update system profile", "refresh profile", "regenerate system profile", etc.

## Inspection procedure

### Step 1: Parse configuration.yaml

Read `/config/configuration.yaml` and extract:

1. **All `!include` directives** — note the integration and target path

   ```yaml
   automation: !include automations.yaml
   script: !include scripts.yaml
   template: !include template.yaml
   ```

2. **All `!include_dir_*` directives** — note the pattern and directory

   ```yaml
   automation: !include_dir_merge_list automations/
   ```

3. **Inline configurations** — integrations configured directly without includes
   ```yaml
   input_boolean:
     guest_mode:
       name: Guest Mode
   ```

### Step 2: Verify what exists

For each path found in Step 1:

- Check if the file or directory actually exists
- Note which are present vs referenced but missing

Also check for these optional directories (only include if they exist):

- `/config/packages/`
- `/config/custom_components/`
- `/config/pyscript/`
- `/config/appdaemon/`
- `/config/python_scripts/`
- `/config/www/`

### Step 3: Detect naming conventions

Sample up to 20 entities across automations, scripts, and input helpers to identify
the conventions this user follows. Look for:

- **Identifier style** — `snake_case`, `camelCase`, `kebab-case`, or mixed
- **Prefixes/namespacing** — e.g. `living_room_*`, `mode_*`, `notify_*`
- **Grouping pattern** — by area, by function, or flat
- **Alias/friendly name template** — look carefully for structural patterns in alias
  names: emoji prefixes, category groupings, separator characters (`:`, `-`, `|`),
  natural language descriptions. If a consistent template is found, document it
  explicitly with an example rather than just describing the casing style.

  Example of what to capture:

  > **Alias template**: `{emoji} {category}: {description}`
  > (e.g. `☀️ garden: irrigation alert`, `🌡️ climate: temperature low`)
  > Use this exact format when creating new automations or scripts.

Use the files already found in Step 2 as the source (e.g. read a few entries from
`automations.yaml` or `scripts.yaml`). If those files don't exist or are empty,
skip this step — do not fabricate conventions.

### Step 4: Detect language

Sample friendly names and aliases from the same files used in Step 3 (automation
`alias:` fields, script names, input helper `name:` fields). Identify the natural
language in use (e.g. English, Catalan, Spanish, French).

**Skip this step entirely if** the user's AGENTS.md or agents-options.md already
specifies a language preference — do not override an explicit setting.

### Step 5: Fetch notification services

Call the HA API to get available notification services:

```bash
curl -s -H "Authorization: Bearer $HA_TOKEN" \
  "$HA_URL/api/services" \
  | jq -r '.[] | select(.domain == "notify") | .services | keys[] | "notify.\(.)"'
```

List only the `notify.*` entries. Skip if the API call fails.

### Step 6: Generate the system-profile skill

Create the directory and skill file:

```bash
mkdir -p /data/pi-agent/agents/skills/system-profile
```

Write `/data/pi-agent/agents/skills/system-profile/SKILL.md` using this template,
populating each section with findings from the steps above. Omit any section where
no data was found.

```markdown
---
name: system-profile
description: >
  Instance-specific profile for this Home Assistant system. Auto-generated by
  system-profile-creator. Read this to know where config files are, what conventions
  the user follows, and what services are available on THIS system.
metadata:
  version: 1
  generated: YYYY-MM-DD
---
```

## Regeneration (when user asks to update)

If system-profile already exists and user explicitly asked to update:

1. Run the full inspection procedure (Steps 1–5)
2. Compare findings with the existing system-profile
3. Show the diff to the user:
   > "Changes detected:
   >
   > - Added: `sensor` now uses `!include sensors.yaml`
   > - Added notify service: `notify.mobile_app_new_phone`
   > - Removed: `input_boolean.yaml` no longer exists
   >
   > Update the profile? (yes/no)"
4. If confirmed, overwrite with the new version
